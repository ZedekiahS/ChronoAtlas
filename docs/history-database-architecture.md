# ChronoAtlas 历史资料数据库架构

ChronoAtlas 后续会覆盖《三国志》《资治通鉴》《后汉书》、裴松之注，以及罗马、萨珊、印度等区域资料。前端 JSON 适合原型，但不适合长期保存多本古籍全文、人物传记、出处段落、关系和审校状态。

当前采用 SQLite 作为本地权威编辑库，前端继续读取导出的轻量 JSON。这样既能保留静态网站的开发速度，也能避免把完整古籍语料打进浏览器 bundle。

## 当前决策

- `db/schema.sql` 定义长期数据库结构。
- `scripts/build-history-db.mjs` 从现有 JSON 构建 `db/chronoatlas.sqlite`。
- `scripts/export-history-data.mjs` 从 SQLite 导出前端可用的切片数据。
- `db/chronoatlas.sqlite` 是构建产物，不提交到 Git。
- 现有 `data/*.json` 暂时仍是可读种子数据；等录入量明显变大后，可以反过来让 SQLite 成为唯一写入源，再导出页面 JSON。

## 为什么现在就做数据库

人物页要展示的不是百科摘要，而是“史料可追踪的一生”。这需要保存：

- 人物基本档案和别名。
- 原始史料来源。
- 原文段落或引用摘录。
- 原文中提到的人物、事件和地点。
- 从原文抽取出的生平节点。
- 人物关系和关系变化时间。
- 哪些传记已经读完，哪些还缺《资治通鉴》或裴注互校。

这些关系用纯前端数组也能做一阵子，但当覆盖到《三国志》所有人物时，查重、审校、来源追踪和增量导出都会变得很痛苦。

## 核心表

### sources

保存史料或现代参考来源。三国第一批来源包括《三国志》各纪传、《资治通鉴》相关卷、《后汉书》相关纪传。罗马扩展时也使用同一张表，例如 Cassius Dio、Herodian、Historia Augusta。

### source_passages

预留给完整古籍段落。以后如果导入《三国志》全文，可以按卷、篇、段落、纪年切分到这里。

### source_mentions

保存“某段原文提到了哪些人物、事件、地点”。它不是最终解释，而是证据层。

### persons 和 person_aliases

`persons` 保存人物稳定档案，`person_aliases` 保存名、字、谥号、庙号、官称、爵号、拉丁名、希腊名等。这样以后处理“太祖”“先主”“吴主”“Imperator Caesar”等称呼时，可以归一到同一个人物。

### person_life_events

人物页的主时间线。每条生平节点都应该连接到 `source_mentions` 或 `sourceRefs`，不能只靠现代摘要。

### historical_events

世界年表和区域对比用的大事件、中事件。它和人物生平分开，因为同一场战争可以关联很多人物，而一个人物也可以在同一年有多个个人节点。

### person_relations

人物关系图。关系有时间范围和来源，不是静态“朋友/敌人”标签。

### coverage_status

审校和覆盖度台账。它回答的问题是：某个人的主传读完了吗？是否已经和《资治通鉴》互校？裴松之注有没有拆分？

## 当前工作流

1. 维护或导入 JSON 种子数据。
2. 运行 `npm run validate:data`，同时校验 JSON 和数据库 schema。
3. 运行 `npm run db:rebuild`，生成本地 SQLite 和前端索引切片。
4. 前端继续读取现有 JSON；后续可逐步改为读取 `data/generated/*` 的切片数据。

## 之后的录入路线

第一阶段先不追求“全文一次性导完”，而是按人物和来源推进：

1. 曹操、刘备、孙权、诸葛亮：主传逐段拆成 `source_mentions`。
2. 袁绍、吕布、陶谦、刘表、刘璋、张鲁、公孙瓒：补齐和官渡、赤壁、益州相关的传记节点。
3. 河北四庭柱、五子良将、蜀汉重臣、江东文武：建立别名和人物覆盖台账。
4. 再按《资治通鉴》卷五十八至八十一做编年互校。

等三国资料链路稳定后，再复用同一 schema 做罗马人物和事件。
